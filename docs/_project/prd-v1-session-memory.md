## 1. 背景与动机（Background）

### 1.1 现状

当前 AEP 协议聚焦于经验的发布、获取和反馈闭环，但存在一个关键缺失：**智能体的行动记录**。

**现有问题：**
1. **数据源错位**：当前设计试图从"日志文件"采集经验，但真正的数据源应该是"智能体与用户的真实交互"
2. **记忆丢失**：每次会话结束后，智能体的上下文丢失，无法形成可复用的记忆
3. **反馈断层**：用户对智能体行为的反馈（显式/隐式）没有被系统性记录
4. **发布延迟**：智能体积累的经验无法及时转化为可发布的 Experience

### 1.2 问题

| 问题 | 影响 | 严重度 |
|------|------|--------|
| 无会话记录机制 | 无法追溯智能体行为，缺乏审计轨迹 | P0 |
| 无标准化行动日志 | Experience 缺乏原始数据支撑 | P0 |
| 反馈与行动脱节 | 无法知道哪个反馈对应哪个具体行为 | P1 |
| 无会话压缩机制 | 长期会话数据膨胀，难以管理 | P1 |
| 无统一存储规范 | 不同工具/环境存储位置不一致 | P2 |

### 1.3 核心洞察

> **数据源应该是"智能体行动记录"而不是"传统日志"**

1. **记忆来自真实交互**：智能体与用户的对话、决策、工具调用才是真正的经验来源
2. **智能体只负责记录**：记录是智能体的职责，但内容由系统整理（参考 OpenClaw 模式）
3. **会话即记忆单元**：每次会话是一个完整的记忆单元，需要被保存、压缩和归档

### 1.4 参考设计

OpenClaw 的会话记忆模式：
- 在 `/new` 时压缩保存会话到 `<workspace>/memory/*.md`
- 会话记录包含：用户意图、智能体行为、结果、反馈

## 2. 目标与成功标准（Goals & Success Metrics）

### 2.1 本 Epic 业务目标

1. **可追溯性**：智能体的每一次行动都有记录，支持审计和复盘
2. **数据完整性**：行动记录与反馈关联，形成完整的经验闭环
3. **存储规范化**：统一存储位置，跨工具/环境可访问
4. **自动化压缩**：长会话自动压缩为摘要，保持可读性

### 2.2 成功标准

| 指标 | 目标 | 口径 |
|------|------|------|
| 会话记录覆盖率 | 100% | 所有智能体交互都被记录 |
| 反馈关联率 | >= 90% | 反馈能够关联到具体 AgentAction |
| 压缩比 | >= 70% | 长会话压缩后体积减少 70% |
| 记录延迟 | < 100ms | 从交互发生到记录完成的时间 |

## 4. 用户与场景（Users & Scenarios）

### 4.1 用户角色

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| **智能体（Agent）** | AI 智能体本身 | 自动记录行动，不遗漏任何交互 |
| **开发者（Developer）** | 集成 AEP SDK 的开发者 | 简单的 API 调用，无需关心存储细节 |
| **审计者（Auditor）** | 需要追溯智能体行为的人员 | 完整的会话记录，可搜索的时间线 |
| **系统（System）** | AEP Hub | 从记录中提取可发布的 Experience |

### 4.2 关键场景

**场景 S1：自动记录交互**
> 智能体与用户进行对话，每次问答、工具调用、决策都被自动记录

**场景 S2：收集用户反馈**
> 用户点击"有帮助"按钮，反馈与对应的智能体回答关联

**场景 S3：压缩归档会话**
> 会话结束后，系统自动将详细记录压缩为摘要，保留关键信息

**场景 S4：发布经验**
> 从会话记录中提取成功的问题-解决方案对，准备发布到 Hub

## 5. 功能需求（Functional Requirements）

### 5.1 F1: 会话自动记录（Session Recording）

**描述**：每次智能体交互（消息、工具调用、决策）自动记录到 JSONL 文件

**业务规则**：
1. 每次交互结束触发记录（不是请求开始）
2. 记录包含完整的上下文信息
3. 支持离线场景（断网时先本地存储）

**触发时机**：
- 用户发送消息 → 智能体回复完成
- 智能体调用工具 → 工具返回结果
- 智能体做出决策 → 决策执行完成

**输出**：
- 文件：`<workspace>/.aep/sessions/session_<timestamp>_<id>.jsonl`
- 格式：每行一个 AgentAction JSON 对象

### 5.2 F2: AgentAction 日志格式（Action Log Format）

**描述**：定义标准化的智能体行动日志格式

**AgentAction 接口**：
```typescript
interface AgentAction {
  // 基础信息
  id: string;                    // UUID，唯一标识
  timestamp: Date;               // 行动发生时间

  // 行动类型
  action_type: 'tool_call' | 'message' | 'decision';

  // 核心内容（参考 Experience 结构）
  trigger: string;               // 遇到什么问题/场景
  solution: string;              // 采取什么行动
  result: 'success' | 'failure' | 'partial';  // 结果

  // 上下文
  context: {
    session_id: string;          // 所属会话 ID
    parent_action_id?: string;   // 父行动 ID（如工具调用的触发消息）
    workspace?: string;          // 工作空间路径
    model?: string;              // 使用的模型
    tools_used?: string[];       // 使用的工具列表
    [key: string]: any;          // 扩展字段
  };

  // 反馈（后续填充）
  feedback?: {
    type: 'explicit' | 'implicit';
    value: 'positive' | 'negative' | 'neutral';
    score?: number;              // 0-1
    source?: string;             // 反馈来源
    timestamp: Date;
  };

  // 元数据
  metadata?: {
    duration_ms?: number;        // 行动耗时
    tokens_used?: number;        // 消耗的 token 数
    confidence?: number;         // 智能体置信度
    [key: string]: any;
  };
}
```

**权限**：智能体自动记录，无需额外授权

**边界条件**：
- 超长内容（>10KB）：截断并标记 `truncated: true`
- 敏感信息：支持标记为 `redacted: true`

### 5.3 F3: 反馈收集机制（Feedback Collection）

**描述**：支持显式和隐式反馈，并与 AgentAction 关联

**反馈类型**：

| 类型 | 触发方式 | 示例 |
|------|----------|------|
| **显式反馈** | 用户主动操作 | 点击"有帮助"按钮、评分 1-5 星 |
| **隐式反馈** | 系统推断 | 用户采纳建议、用户重新提问（暗示不满意） |

**关联机制**：
1. 每个 AgentAction 有唯一 ID
2. 反馈提交时携带 `action_id`
3. 系统将反馈追加到对应 AgentAction

**业务规则**：
- 显式反馈优先级高于隐式反馈
- 同一行动的多次反馈，取最新值
- 隐式反馈需要设置合理的推断阈值

### 5.4 F4: 会话压缩与归档（Session Compression & Archive）

**描述**：定期压缩会话记录，生成可读摘要

**压缩触发条件**：
1. 会话结束（用户关闭或超时）
2. 会话记录超过阈值（如 100 条行动）
3. 手动触发压缩

**压缩策略**：
```
原始会话 (100条 AgentAction)
    ↓ 压缩
摘要 (关键信息保留)
    - 用户意图 (1-2 句)
    - 主要问题 (3-5 个)
    - 成功的解决方案 (2-3 个)
    - 失败尝试 (1-2 个，含教训)
    - 用户反馈总结
    - 总耗时 / token 消耗
```

**归档位置**：
- 原始记录：`<workspace>/.aep/sessions/archive/`
- 压缩摘要：`<workspace>/.aep/memory/session_<timestamp>_summary.md`

### 5.5 F5: 统一存储位置（Unified Storage Location）

**描述**：定义所有 AEP 相关数据的统一存储结构

**目录结构**：
```
<workspace>/.aep/
├── agent.json              # 智能体身份信息
├── sessions/               # 会话记录
│   ├── session_<ts>_<id>.jsonl    # 当前活跃会话
│   └── archive/                   # 已归档会话
│       └── session_<ts>_<id>.jsonl.gz
├── memory/                 # 压缩记忆
│   ├── 2026-02-23_summary.md      # 每日摘要
│   └── session_<ts>_summary.md    # 会话摘要
├── pending/                # 待发布队列
│   ├── exp_<id>.json              # 待发布的 Experience
│   └── batch_<ts>.json            # 批量发布队列
└── cache/                  # 本地缓存
    ├── experiences.json           # 获取的经验缓存
    └── signals.json               # 信号索引缓存
```

## 7. 业务约束与风险（Constraints & Risks）

### 7.1 用户体验要求

* 响应速度：记录操作不阻塞主流程，延迟 < 100ms
* 存储空间：默认保留最近 30 天会话，超过自动归档
* 可读性：压缩后的摘要人类可读，支持快速回顾

### 7.2 错误处理

* 写入失败：重试 3 次，失败后放入内存队列，下次会话恢复
* 磁盘空间不足：提示用户清理，停止新记录
* 文件损坏：跳过损坏行，记录错误日志

### 7.3 权限和隐私

* 会话记录存储在用户本地工作空间
* 敏感信息（密码、密钥）支持标记为 `redacted`
* 上传到 Hub 前需用户确认（或设置自动发布规则）

### 7.4 业务约束

* 默认开启自动记录，用户可选择关闭
* 压缩归档默认在会话结束后自动执行
* 待发布队列支持手动管理和自动发布

### 7.5 业务风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 磁盘空间占用大 | 用户体验差 | 自动清理策略、压缩归档 |
| 敏感信息泄露 | 安全风险 | 敏感标记、发布前确认 |
| 记录丢失 | 数据不完整 | 异步写入、内存队列兜底 |
| 反馈关联错误 | 数据不准确 | 唯一 ID、时间窗口校验 |
